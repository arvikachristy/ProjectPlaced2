In this section, we are going to introduce all of our platform features as well as introducing the system and user manuals. We will also mention all of the tools that have we used to build the particular sections, and the justification for why we have included each feature.

System Manual

Compatible Devices
Most recent Operating Systems are compatible to run the server side of our platform. This includes:

  • Windows

    Windows 8 (Desktop), Windows 7, Windows Vista SP2, Windows Server 2008, Windows Server 2012 (64-bit)

  • MAC OS X

    Mac OS X 10.7.3 or later

  • Linux

    Oracle Linux 5.5+, Oracle Linux 6.x (32-bit), 6.x (64-bit), Oracle Linux 7.x (64-bit), Red Hat Enterprise Linux 5.5+, 6.x (32-bit), 6.x (64-bit), Ubuntu Linux 10.04 or later

Connectivity
To run the server properly, make sure the device has a good connection to the internet. The page may not display properly without an internet connection.

Browser Support
Most browsers will support this platform. However, our frontend is built from AngularJS and the Ionic Framework hence the most updated version of the web browser is highly recommended. Here is the list of our recommended web browsers:
  • Safari
  • Google Chrome
  • Mozilla Firefox
  • Opera

User Manual

Run the system configurations
In order to set up the Project Placed Team Management tool, there are two main parts of the system; the client and the server. Make sure you are connected to the internet and through the system terminal, we will start by connecting to the server.
To set up the server:
  • go to the file directory where we have placed the manage.py file.
  • Type python manage.py runserver in order to run the main server of the project.


Run the platform
To run the platform itself, we are going to set up the platform using Google Chrome with the localhost 8000 as the example. To do this, just type the following URL on your browser localhost:8000/trackprogress and the platform should show up.

Platform Features

There are two versions of our main page. The page seen above is the student version, which the students will use to manage their project. There is also another version of the page which the supervisor will use to check how the project is going (without having the ability to add to or modify any of the features on the students' page). In our platform, you can switch between these two pages by clicking on the button on the left of the header.



The Project Banner
On the project banner, you can see the name of the company and project, as well as a small description about what the project is about. There is also an image in the middle, which can be the logo of the company/project. The banner would be useful to people who are checking the page without checking previous pages related to the project which explain what the project is about. E.g. the supervisor may have a lot of projects to manage, so it could be useful to have some brief background about the project to remind them what project the page is refering to.

User Visibility + Activities
There are icons which display the users who are part of the project. Underneath that, there is the button which leads to the Gantt chart page, and the button which opens the comment box modal. The project placed frontend already includes the user visibility icons, so we can include it in our page as well. We are using project placed CSS, so our page can be consistent with the already existing designs.



Status Section
There is a radio button with status options. The students can choose the status option they feel best represents the current status of their project. This choice will remain on their project page until the students choose a different choice. Each choice is a different colour - either green, amber or red. There is also some text, describing what the feature is for.

When interviewing a teaching assistant and the supervisor of the projects, we heard from them that they want to help assist the students with their projects as much as possible. However, it was very difficult for them to identify when projects were having difficulties. We also used the red, amber, green philosophy we had previously looked into. We also believe it is important for students to know exactly what this feature is for (which is why we have some desscription letting them know). We don't want them to feel discouraged about putting their project with major issues because that will relect poorly on them, but encouraged to select that project so they can recieve help with it.

We used Angular Material - a UI Component framework. We used it to give the radio buttons an appealing design. It worked well with our project as we are using Angular for the front end, and was easy to implement.




On the supervisor page, the supervisor (or TA/client etc) may give their own opinion of the status (so the status of the students and the status of the teaching assistants are separate. On the supervisor page, we have implemented a table which tracks the changes in the status options chosen by the students and those chosen by the supervisor. We believe that this shows an overview of how the project is going by tracking changes over time. The chart has been implemented with Django Chartit.

Gantt Chart
From the main page, we can access the Gantt Chart section. This is the page with the gantt chart on it. There are functions to search and filter the tasks as well as the ability to specify the date range you would like to view. It makes sure the development of the project is on time and can be viewed based on timeline, milestones or sprints. You can customize the taskss to be done by dragging them around (to improve user experience) and placing them on the desired date. Dependencies between tasks are also available to make sure the user does not have to deal with re-organizing the deadlines.

We wanted a method of tracking the tasks/ plans of the project over a period of time. We've temporarily implemented a gantt chart because in our own projects, we have used the gantt charts many times to display the development plan, and it was good to include a system that students are already accustomed to. However, in the future, the gantt chart can be replaced with any other methods of tracking progress.

This is using Angular Gantt (which is a gantt chart component for AngularJS). We've stripped and modified some features from the original Angular Gantt - keeping the functionality but making sure users aren't confused by the large selection of options.



Comment Box
Another feature is the comment box. When you click on the comment box button, a modal opens. In this modal, the student can write their name, as well as their message, and post it for others to see. From our experiencies working with teams with the past at university, we believe that communication is a very key point in team projects. Our interviews with other students also told us that they felt the same way, and bad communication was one of the big problems that they sometimes faced. We decided that to have a comment box on the page which would allow all people associated with the project to leave comments and feedback that will be seen by other members.


Request Resources
Under the status radio buttons, there is a button which opens a modal. In this modal, the students can send a request to their supervisor/ teaching assistant requesting any resources that they require. They can select (from a selection of options) exactly what resource they require and how urgently they need it, as well as leave a comment with more details. From our interview with the supervisor/ TA we heard that their experience with projects is that many times students don't ask for resources from them, even though they really require it. By including a request resources option, we want to encourage students to be more vocal if they need anything, and we want to make it easier for the supervisor to be aware of exactly what resources each project needs.

We also used Angular material for the radio buttons, since we had already implemented it in the status section. Everything is contained within an ionic modal (which is opened by an appropriate controller).



Task Table
The students can add, edit, and delete the tasks they are working on to a task table on the page. They can also search through the task list with keywords. This task table has been implemented to help students organise their work (especially when tasks are due, and who they are assigned to), as well as letting the supervisor/TA/client have an overview of all the tasks being worked on, so they can see if the team is on track.



Bi-Weekly Index
On the index page, the biweekly report section is split into two parts. On the left pane, you can see the current biweekly report being worked on, in a calender form. You can view and edit the report or submit it. Before submitting, a modal appears asking for confirmation, since reports can not be edited once they are submitted. On the right pane, you can see a list of the previous reports that have already been submitted. Clicking on one of the previous reports will take you to a page where you can view that report (but not make any changes since the page has already been submitted).

We had discussions about the best way to display the information (of what the current report is and how we would display the previous reports). We started with the idea of displaying a carousel style report but decided that it will not be a good choice since there could potentially be many reports to cycle through and we only wanted to represent the biweekly reports without taking too much space. So we replaced it with a list of the reports.



Bi-Weekly Reports Page
On the biweekly page, there are different fields that the students will fill out for their report, which will then be submitted for the clients, supervisors and TAs to see. The current report they are working will not be able to be seen by the clients/supervisors. They are free to make as many changes as they want (saving those changes), but will not be able to make any changes to that specific report once submitted.

Students already have to create biweekly reports when working on projects. The aim of these reports is to give an overview of the completed tasks, meetings, and any issues or challenges faced. However, at the moment there is no set template to make things consistent. We believe that having a consistent template for the reports would be useful. The students would be more aware of the information they should feedback, and it would be easier for teaching assistants to look through many projects since they have a consistent format. Last term, during our research and data collection, we gathered biweekly reports from students and analysed them to see the way that students had organised their reports. We created a set template from this.
